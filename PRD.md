# gxk的待办清单 (进阶版) - 产品需求文档 (PRD)

## 1. 产品概述 (Product Overview)

**1.1 产品背景** 本项目旨在开发一款高效、清爽、无广告的个人任务管理工具。基于现代化前端技术栈，在提供基础的待办事项增删改查功能之上，加入多维度筛选、数据统计展示以及便捷的天气信息获取功能，提升个人日常时间管理的效率。

**1.2 核心目标**

- 提供极简且现代化的卡片式双栏 UI 操作体验。
- 实现健壮的本地数据持久化，确保刷新不丢失。
- 提供多条件组合筛选与模糊搜索功能，便于任务回溯。
- 预留良好的架构扩展性，便于后期接入云端同步或子任务功能。

**1.3 技术栈选型**

- **前端框架:** React + TypeScript + Vite
- **状态管理:** Zustand
- **样式方案:** Tailwind CSS
- **日期处理:** date-fns
- **数据存储:** Web LocalStorage
- **图标库:** Lucide-React

------

## 2. 功能架构 (Functional Architecture)

包含四大核心模块与一个外部扩展模块：

1. **任务管理模块：** 任务的创建、编辑、状态切换、删除。
2. **列表与视图模块：** 任务卡片展示、优先级与分类视觉区分。
3. **数据中心模块：** 多条件过滤（分类/优先级/状态）、关键词模糊搜索、任务完成度统计。
4. **持久化模块：** 基于 LocalStorage 的数据读写封装。
5. **扩展模块：** 实时天气小组件（支持城市切换）。

## 3. 详细功能需求 (Detailed Features)

### 3.1 全局布局与 UI 规范

- **色彩规范：** 背景色为米黄色（`#f5f5f0`），主体卡片为纯白色并带有细边框与极浅阴影。
- **页面布局：**
  - **顶部 Header：** 左侧显示主标题“鱼皮的待办清单”及副标题，右侧放置“天气组件”。Header 下方有一条贯穿的细黑线。
  - **主体内容区：** 采用 3:7 的左右分栏布局。左侧为“数据面板”，右侧为“操作与列表面板”。

### 3.2 核心业务功能

#### 3.2.1 统计数据面板 (左侧上方)

- **展示项：** - `总任务`：当前所有任务数量。
  - `待完成`：未勾选的任务数量（数字使用橙色高亮）。
  - `已完成`：已勾选的任务数量（数字使用绿色高亮）。
  - `完成率`：(已完成 / 总任务) * 100%，保留整数。
- **逻辑：** 随全局状态 `todos` 的变化实时重新计算。

#### 3.2.2 分类筛选面板 (左侧下方)

- **UI：** 垂直排列的按钮组。包含：“全部任务”、“工作”、“学习”、“生活”。
- **交互：** 点击对应分类时，按钮高亮显示为深色背景，同时右侧任务列表实时过滤出对应分类的任务。

#### 3.2.3 快速添加任务 (右侧上方)

- **UI：** 在一个白色卡片内水平排列的表单。
- **输入项：**
  - `任务标题` (Input，必填，Placeholder: "输入任务内容...")。
  - `分类` (Select，单选："工作", "学习", "生活")。
  - `优先级` (Select，单选："高", "中", "低")。
  - `添加按钮` (深灰色背景，点击触发保存)。
- **进阶逻辑（弹窗/展开区）：** 允许用户进一步填写 `描述 (Description)` 和选择 `截止日期 (DueDate)`。

#### 3.2.4 任务列表与搜索 (右侧下方)

- **头部区：** 左侧标题为“今日任务”或当前所选分类名，右侧为一个搜索框（支持根据标题和描述进行**模糊搜索**）。
- **任务卡片展示：**
  - 左侧：复选框（点击切换 `completed` 状态，勾选后标题增加删除线样式，文字变灰）。
  - 中间：主标题（上）、副标题/描述摘要（下）。
  - 右侧：分类标签（带浅色边框）、优先级标签（高：红/橙色；中：蓝色；低：灰色）、截止日期（使用 `date-fns` 格式化为 `YYYY-MM-DD`）。
- **操作：** 鼠标悬浮任务项时，右侧显示“编辑”和“删除”小图标。

#### 3.2.5 天气扩展组件 (右上角)

- **UI：** 小型卡片或纯文本+图标形式。
- **功能：**
  - 默认展示用户预设城市的天气图标和温度。
  - 支持点击输入框修改城市名称，调用免费第三方天气 API（如和风天气、OpenWeather 等）重新获取并渲染数据。

## 4. 数据模型设计 (Data Model)

所有的核心数据必须严格遵循以下 TypeScript 接口定义：

TypeScript

```
// src/types/todo.ts

export type Category = '工作' | '学习' | '生活' | string;
export type Priority = '高' | '中' | '低';

export interface Todo {
  id: string;               // 唯一标识 (UUID 或时间戳)
  title: string;            // 任务标题 (必填)
  description?: string;     // 任务描述 (选填)
  category: Category;       // 分类
  priority: Priority;       // 优先级
  dueDate?: string;         // 截止日期 (ISO 格式字符串)
  completed: boolean;       // 是否已完成 (默认 false)
  createdAt: string;        // 创建时间 (ISO 格式字符串)
}

export interface TodoFilter {
  category: string | 'all'; // 筛选分类，'all' 表示全部
  priority: Priority | 'all';
  status: 'all' | 'active' | 'completed';
  searchQuery: string;      // 搜索关键词
}
```

## 5. 数据流与持久化要求 (Data Flow & Persistence)

1. **状态管理 (Zustand)：**
   - 必须创建一个统一的 Store (`useTodoStore`)，集中管理 `todos` 数组和 `filter` 对象。
   - 所有对任务的增（`addTodo`）、删（`deleteTodo`）、改（`updateTodo`）、状态切换（`toggleTodo`）操作，必须作为 Store 的 Actions 存在。
2. **持久化 (LocalStorage)：**
   - 每次 Store 中的 `todos` 状态发生改变时，必须自动将其序列化并保存至 LocalStorage。
   - 首次加载应用时，Store 的初始状态需从 LocalStorage 中反序列化读取，若无数据则设为空数组。

## 6. 未来演进路线 (Future Roadmap) - 预留思考

1. **多标签系统：** 从单分类模式演进为 `tags: string[]` 的多标签模式。
2. **子任务支持：** 在 `Todo` 模型中加入 `subTasks: SubTask[]` 字段。
3. **云端同步：** 预留 API 接口层，未来可将 LocalStorage 的读写替换为 Firebase 或自建 Node.js 接口的 HTTP 请求。
